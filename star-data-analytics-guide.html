Q1: Describe a time you troubleshot a complex issue across multiple Azure data services.

S A production data pipeline spanning ADLS → Databricks → Azure SQL DWH was failing intermittently during nightly loads. The pipeline processed 50M+ records from multiple source systems and failures were impacting downstream Power BI dashboards used by executives.

T I needed to identify the root cause across the multi-service chain, restore data freshness, and prevent recurrence — all under SLA pressure with executive visibility.

A I:

  • Mapped the end-to-end data flow: ADLS landing zone → Databricks Bronze/Silver/Gold layers → COPY INTO Azure SQL DWH → Power BI refresh
  • Checked Azure Monitor and Databricks cluster logs — found intermittent java.lang.OutOfMemoryError during the Silver→Gold transformation
  • Analyzed the Spark UI: a single partition had 80% of the data (severe data skew on customer_region key)
  • Cross-referenced with ADLS file sizes — one source system had changed from daily to hourly dumps without notification, causing 8x volume spike in one partition
  • Implemented salting technique on the skewed key and added adaptive query execution
  • Added file-count and size-based alerting on the ADLS landing zone to catch upstream changes

R Pipeline stabilized with zero failures over next 30 days. Processing time reduced from 4.5 hours to 1.8 hours. The monitoring alert caught 2 additional upstream schema changes before they caused failures.

L In multi-service architectures, the root cause is often NOT where the error manifests. Systematic tracing from symptom backwards through each service boundary is critical. Upstream change detection is as important as downstream error handling.

Q2: How do you approach Azure service limits and throttling issues in data workloads?

S A large dataset upload process was failing with HTTP 429 (throttling) errors when writing to Azure SQL Database. The application was batch-loading 200M records from ADLS via multiple parallel streams.

T I needed to diagnose why throttling occurred and redesign the load pattern to work within Azure service limits while meeting the 4-hour SLA window.

A I:

  • Reviewed Azure SQL DTU/vCore consumption via sys.dm_db_resource_stats — confirmed we were hitting 100% CPU during parallel loads
  • Analyzed the parallelism: 32 concurrent COPY streams was overwhelming a Standard S3 tier
  • Reduced parallelism to 8 streams but added batch windowing (load by date partition)
  • Switched from row-by-row inserts to BULK INSERT with TABLOCK for minimal logging
  • Implemented exponential backoff retry logic in the Python orchestrator
  • Recommended tier upgrade to Hyperscale for sustained large workloads

R Eliminated throttling errors completely. Load time improved from 6 hours (with failures) to 3.2 hours. Cost increased only 15% with tier adjustment vs. 4x if we had simply scaled massively.

L Throwing more parallelism at Azure services without understanding DTU/throughput limits creates worse outcomes. Right-sizing concurrency + optimal bulk patterns often beats brute-force scaling.

⚡ DATABRICKS & PYSPARK

Q4: Walk me through troubleshooting a slow PySpark job in Databricks.

S A PySpark transformation job processing 500GB of Delta tables was taking 6+ hours — well beyond the 2-hour SLA. The job reads from multiple Delta tables, performs joins and aggregations, and writes to a Gold layer table used for reporting.

T I needed to identify performance bottlenecks and optimize the job to meet the 2-hour window without increasing cluster cost significantly.

A I:

  • Opened the Spark UI → Stages tab: found one stage taking 80% of total time with massive shuffle write (200GB)
  • Drilled into the SQL tab → physical plan showed a SortMergeJoin on a highly skewed key
  • Checked partition distribution: df.groupBy("join_key").count().orderBy(desc("count")) — confirmed 90% of records in 5 partitions
  • Applied broadcast join for the smaller table (< 1GB): broadcast(df_small)
  • For the large-large join: implemented salting — appended random suffix to skewed keys
  • Enabled Adaptive Query Execution: spark.conf.set("spark.sql.adaptive.enabled", "true")
  • Added ZORDER BY on frequently filtered columns in Delta tables
  • Replaced Python UDFs with native Spark functions (when, coalesce, regexp_extract)

R Job runtime dropped from 6.2 hours to 1.4 hours (77% improvement). Shuffle reduced from 200GB to 12GB. No cluster size increase needed — same cost, 4x faster.

L In PySpark, 90% of performance issues come from 3 things: data skew, unnecessary shuffles, and Python UDFs. Always check the physical plan before changing cluster config. AQE + broadcast hints solve most join performance issues.

Q5: How do you handle schema evolution issues in Delta Lake?

S A source system added 3 new columns to their daily feed without advance notice. The Delta table write failed with AnalysisException: cannot resolve column because schema enforcement rejected the new columns.

T I needed to restore the pipeline immediately and implement a sustainable schema evolution strategy.

A I:

  • Immediate fix: added .option("mergeSchema", "true") to the write operation
  • Verified new columns didn't conflict with existing ones using DESCRIBE HISTORY
  • Implemented a schema validation layer that compares incoming schema vs. expected schema BEFORE writing
  • Created alerting: if schema drift detected, log the diff and notify the team (but allow additive changes)
  • Set table property: delta.columnMapping.mode = 'name' for safer evolution
  • Documented schema contract with source team — breaking changes require 48-hour notice

R Pipeline restored within 30 minutes. Schema validation caught 4 additional schema changes over the next quarter — all handled gracefully without failures. Zero downstream impact.

L Schema evolution is inevitable in production. The question isn't "if" but "how gracefully." Additive changes should auto-merge; breaking changes need contracts. Delta's schema enforcement is a feature, not a bug — it prevents silent data corruption.

🗃️ T-SQL & DATA WAREHOUSE

Q7: Describe a time you tuned a critical production query that was causing SLA breaches.

S A nightly stored procedure that aggregates sales data across 15 fact tables was running for 8+ hours, causing the reporting SLA (data available by 6 AM) to be breached 3 days in a row. The procedure processes 2B+ rows.

T I needed to reduce execution time to under 3 hours to restore SLA compliance while ensuring data accuracy.

A I:

  • Captured the actual execution plan — found 4 table scans on billion-row tables and 2 implicit conversions
  • Identified parameter sniffing: the plan was optimized for a small date range but used for full monthly aggregation
  • Broke the monolithic procedure into stages using temp tables: 
    -- Stage 1: Pre-aggregate by date partition
SELECT date_key, product_key, SUM(amount) as total
INTO #stage1
FROM fact_sales WITH (NOLOCK)
WHERE date_key BETWEEN @start AND @end
GROUP BY date_key, product_key;

-- Create index on temp table
CREATE CLUSTERED INDEX ix_stage1 ON #stage1(date_key, product_key);

-- Stage 2: Join with dimensions
SELECT s.*, d.region, p.category
INTO #stage2
FROM #stage1 s
JOIN dim_date d ON s.date_key = d.date_key
JOIN dim_product p ON s.product_key = p.product_key;
  • Added missing non-clustered indexes on join columns with INCLUDE for covering
  • Fixed implicit conversions: VARCHAR to NVARCHAR comparisons causing scans
  • Added OPTION (RECOMPILE) for the date-range-dependent sections
  • Implemented partition elimination by aligning the WHERE clause with table partitioning

R Execution time dropped from 8.2 hours to 1.8 hours (78% improvement). SLA restored immediately. The staged approach also made debugging easier — each temp table is a checkpoint.

L Large procedures benefit from decomposition. Temp tables with indexes create "materialized checkpoints" that give the optimizer better cardinality estimates. Always check for implicit conversions first — they're silent killers that force full scans.

Q8: How do you analyze execution plans to find performance issues?

Key things I look for in execution plans:

SymptomRoot CauseFix
Table Scan on large tableMissing index or non-SARGable WHERECreate covering index; rewrite predicate
Hash Match (high cost)Large unsorted join inputsAdd index on join key; reduce input set first
Sort operator (spills to disk)Insufficient memory grantAdd ORDER BY-aligned index; increase memory grant hint
Thick arrows (actual > estimated)Stale statisticsUPDATE STATISTICS; use FULLSCAN for critical tables
Parallelism with skewOne thread doing all workRedistribute data; adjust MAXDOP; partition table
Key Lookup + Nested LoopNon-covering indexAdd INCLUDE columns to the index
-- Quick diagnostics query pack
-- 1. Find missing indexes
SELECT * FROM sys.dm_db_missing_index_details WHERE database_id = DB_ID();

-- 2. Find expensive queries
SELECT TOP 10 
    qs.total_elapsed_time/qs.execution_count as avg_elapsed,
    qs.execution_count,
    SUBSTRING(qt.text, qs.statement_start_offset/2, 100) as query_text
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) qt
ORDER BY qs.total_elapsed_time DESC;

-- 3. Check index usage
SELECT OBJECT_NAME(i.object_id), i.name, 
    us.user_seeks, us.user_scans, us.user_lookups
FROM sys.indexes i
LEFT JOIN sys.dm_db_index_usage_stats us ON i.object_id = us.object_id
WHERE us.database_id = DB_ID();

Q10: How would you troubleshoot deadlocks in a production DWH?

S Production ETL loads were failing intermittently with deadlock errors. The loads run concurrent updates on dimension tables while reporting queries read the same tables.

T Eliminate deadlocks without impacting either ETL SLAs or reporting query performance.

A I:

  • Enabled deadlock trace flag 1222 and captured deadlock graphs from system_health extended events
  • Analyzed the graph: ETL held page lock on dim_customer while waiting for dim_product; reporting held dim_product waiting for dim_customer (classic cycle)
  • Implemented READ_COMMITTED_SNAPSHOT isolation for the reporting database to eliminate reader-writer conflicts
  • Standardized lock ordering in ETL procs: always acquire locks in alphabetical table order
  • Added NOLOCK hints for reporting queries where slight staleness is acceptable
  • Reduced transaction scope in ETL: commit per-dimension rather than all-at-once

R Deadlocks eliminated (from 15/week to 0). ETL reliability improved to 99.9%. Reporting queries actually got 20% faster due to RCSI eliminating blocking waits.

L Deadlocks are design problems, not runtime problems. RCSI is almost always the right choice for mixed OLTP/reporting workloads. Lock ordering discipline prevents cycles at the source.

Q14: Tell me about a time a production ETL pipeline failed and how you resolved it.

S A critical nightly ETL pipeline that loads financial transaction data from 12 source systems into the data warehouse failed at 2 AM. The pipeline processes 80M records and feeds 15 downstream reports due by 7 AM. The on-call analyst escalated to me after initial troubleshooting showed no obvious errors in the ADF logs.

  • Investigated the source: one of 12 systems (SAP extract) had a file 10x larger than normal (schema change added nested records)

L In production support, "perfect" is the enemy of "on time." Loading 11/12 sources on time is better than loading 12/12 sources late. Always have a partial-load strategy. Schema validation at ingestion is non-negotiable for multi-source pipelines.

Q16: How do you handle data pipeline idempotency?

Answer: Every pipeline I build must be safely re-runnable. Key patterns:

  • Upsert/MERGE: Use MERGE INTO target USING source ON key WHEN MATCHED THEN UPDATE WHEN NOT MATCHED THEN INSERT
  • Partition overwrite: Delete-then-insert by partition key (date), not append-only
  • Watermark tracking: Store last-processed timestamp; on re-run, start from watermark
  • Delta Lake: Use MERGE with business keys — naturally idempotent
  • File-based: Process files atomically — move to "processed" folder only after successful load

Q19: How do you handle large Parquet file optimization?

Key optimization strategies:

  • File size targeting: Aim for 128MB-1GB per file (optimal for Spark readers)
  • Too many small files: Use OPTIMIZE (Delta) or repartition + coalesce
  • Column pruning: Parquet's columnar format means SELECT specific columns = dramatic I/O reduction
  • Predicate pushdown: Ensure filters on partition columns to skip entire file groups
  • Compression: Use SNAPPY for balanced speed/size, ZSTD for maximum compression
  • Row group size: Default 128MB is usually optimal; tune for query patterns

Q20: How do you implement incremental loads in ADF?

S A full-load pipeline was taking 12 hours to reload 500M records nightly from an operational database. Storage costs were growing 20% monthly due to full snapshots.

T Convert to incremental load pattern to reduce time, cost, and source system impact.

A I designed a watermark-based incremental pattern:

Pipeline Structure:
1. Lookup Activity → Read last_watermark from control table
2. Copy Activity → Source query: 
   SELECT * FROM source_table 
   WHERE modified_date > '@{activity('Lookup').output.firstRow.watermark}'
3. Stored Procedure → Update control table with MAX(modified_date)
4. If Activity → If row_count > 0, trigger downstream transformations
  • For file-based sources: used event triggers (BlobCreated) on ADLS
  • For CDC sources: leveraged sys.change_tracking_current_version()
  • Added error handling: if Copy fails, watermark is NOT updated (safe replay)
  • Implemented tumbling window triggers for guaranteed processing windows

R Load time reduced from 12 hours to 45 minutes (94% improvement). Storage costs reduced by 60%. Source system load reduced by 85% (only reading changed records).

L Incremental loads are more complex to implement but pay off exponentially as data grows. The watermark pattern with atomic commit-only-on-success ensures data consistency even during failures.

Q21: How do you troubleshoot ADF pipeline failures?

My systematic approach:

  • Step 1 — ADF Monitor: Check pipeline run → identify failed activity → read error message
  • Step 2 — Error categorization:
    • Connectivity: timeout, auth failure → check linked service, network, firewall
    • Data: schema mismatch, type conversion → check source data changes
    • Resource: memory, DTU limit → check cluster/database sizing
    • Logic: wrong results → debug mapping data flows in debug mode
  • Step 3 — Integration Runtime logs: For self-hosted IR, check event viewer on IR machine
  • Step 4 — Reproduce: Re-run with same parameters; use Debug mode for step-through
  • Step 5 — Fix + Prevent: Apply fix, add retry policy, alert on this error pattern

Q22: How do you handle ADF data flow performance issues?

Performance optimization checklist:

  • Use staging for COPY activities (enables PolyBase/COPY INTO for DWH loads)
  • Set appropriate parallelism (DIU for Copy, core count for Data Flows)
  • Enable partition option in sources — hash partition on join keys
  • Avoid row-by-row lookups in data flows — pre-load lookup tables as cached datasets
  • Use TTL (Time-to-Live) on data flow clusters to avoid cold-start overhead
  • Monitor Spark execution plan in data flow monitoring for skew/spill indicators

Q23: Describe your experience with SSIS package troubleshooting.

S A legacy SSIS package that loads daily HR data into the data warehouse was failing intermittently with "buffer full" errors. The package processed 5M employee records with multiple lookups and conditional splits.

T Stabilize the package without a full rewrite (legacy system, minimal dev budget).

A I:

  • Enabled SSIS logging (OnError, OnWarning, BufferSizeTuning events) to identify the bottleneck
  • Found: Lookup transformations were using Full Cache mode loading 50M reference rows into memory
  • Switched large lookups to Partial Cache with appropriate cache size limits
  • Identified a Slowly Changing Dimension component causing row-by-row updates — replaced with staging table + MERGE pattern
  • Adjusted DefaultBufferMaxRows and DefaultBufferSize properties based on average row width
  • Split the monolithic data flow into 3 sequential phases with checkpoint restartability

R Package runtime stabilized from "fails 3x/week" to zero failures over 6 months. Runtime improved from 4 hours to 1.5 hours. Memory usage dropped 60%.

L SSIS performance issues are almost always memory-related. Full Cache lookups against large tables are the #1 offender. The SCD wizard component should never be used for tables over 100K rows — always prefer staging + MERGE.

Q29: How do you handle data skew in distributed processing?

Data skew detection:

# PySpark: Detect skew
df.groupBy("partition_key").count() \
  .orderBy(desc("count")) \
  .show(10)

# If max_count / avg_count > 10x → severe skew

Mitigation strategies:

  • Salting: Append random suffix to skewed key, join with exploded version of other table
  • Broadcast join: If one side fits in memory (< 1GB), broadcast it
  • Adaptive Query Execution (AQE): Spark 3.0+ auto-handles some skew scenarios
  • Custom partitioning: Repartition by a more evenly distributed key
  • Two-pass approach: Process skewed keys separately with higher parallelism
# Salting technique
from pyspark.sql.functions import concat, lit, floor, rand

# Add salt to skewed table
df_skewed = df_skewed.withColumn("salt", floor(rand() * 10))
df_skewed = df_skewed.withColumn("salted_key", concat("join_key", lit("_"), "salt"))

# Explode the other table
from pyspark.sql.functions import explode, array
df_other = df_other.crossJoin(
    spark.range(10).withColumnRenamed("id", "salt")
).withColumn("salted_key", concat("join_key", lit("_"), "salt"))

# Join on salted key — evenly distributed
result = df_skewed.join(df_other, "salted_key")

Q31: How do you handle financial data reconciliation between ERP and DWH?

Reconciliation framework for financial data:

  • Level 1 — Row counts: Total documents in source vs. target (should match within delta extraction window)
  • Level 2 — Control totals: Sum of amounts at GL account level must balance between systems
  • Level 3 — Trial balance: Debits = Credits check in DWH (fundamental accounting equation)
  • Level 4 — Sample audit: Random selection of 50 documents, trace from source → staging → DWH end-to-end
  • Level 5 — Exception reporting: Flag any transactions that exist in one system but not the other
-- Financial reconciliation SQL
-- Compare GL balance between SAP extract and DWH
SELECT 
    s.gl_account,
    s.source_balance,
    d.dwh_balance,
    s.source_balance - d.dwh_balance as variance,
    CASE WHEN ABS(s.source_balance - d.dwh_balance) > 0.01 
         THEN 'INVESTIGATE' ELSE 'OK' END as status
FROM sap_trial_balance s
FULL OUTER JOIN dwh_trial_balance d ON s.gl_account = d.gl_account
WHERE ABS(ISNULL(s.source_balance,0) - ISNULL(d.dwh_balance,0)) > 0.01
ORDER BY ABS(s.source_balance - d.dwh_balance) DESC;

🧩 SCENARIO-BASED QUESTIONS

Q38: A user reports that their dataset upload is stuck at 95%. What do you check?

  • Network: Large file upload timeout? Check if connection was dropped (partial upload in ADLS?)
  • File validation: Is there a post-upload validation step that's hanging? (schema check, virus scan)
  • Lock contention: Is the target table locked by another process? Check sys.dm_tran_locks
  • Transaction log: Is the transaction log full? Check DBCC SQLPERF(LOGSPACE)
  • Memory/TempDB: Sort operations spilling to disk during bulk insert?
  • Index rebuilds: Disabled indexes being rebuilt after bulk insert? (common with SSIS bulk loads)
  • Trigger: Is there an AFTER INSERT trigger processing all 95% of rows?

My approach: Check system DMVs for active requests, wait types will tell you exactly what it's waiting on. RESOURCE_SEMAPHORE = memory wait, WRITELOG = log disk I/O, LCK_M_X = blocking.

Q39: Production data shows duplicates after a pipeline re-run. How do you investigate and fix?

Investigation:

  • Identify the duplication scope: How many duplicates? Which tables? Which run IDs?
  • Check pipeline idempotency: Was the pipeline designed for safe re-run? (MERGE vs. INSERT)
  • Check for missing WHERE clause in the re-run (full load instead of incremental)
  • Verify watermark: Did the watermark reset or fail to update from the first run?

Immediate fix:

-- Deduplicate using ROW_NUMBER
WITH ranked AS (
    SELECT *, ROW_NUMBER() OVER (
        PARTITION BY business_key 
        ORDER BY load_timestamp DESC
    ) as rn
    FROM target_table
    WHERE load_date = '2026-06-11'
)
DELETE FROM ranked WHERE rn > 1;

-- Verify
SELECT business_key, COUNT(*) 
FROM target_table 
GROUP BY business_key 
HAVING COUNT(*) > 1;  -- Should return 0 rows

Prevention: Implement MERGE pattern, add unique constraints on business keys, use Delta Lake's MERGE for natural idempotency.

Q40: A Databricks job succeeds but produces wrong results. No errors in logs. How do you debug?

  • Version comparison: Use Delta time travel to compare output before/after the suspect run
  • Input validation: Check if source data changed (new values, nulls, format changes)
  • Logic walkthrough: Step through notebook cell-by-cell with a small sample dataset
  • Join analysis: Check for fan-out (1-to-many joins creating duplicates) or data loss (inner join dropping unmatched rows)
  • Filter context: Did a WHERE clause accidentally exclude valid data?
  • Type coercion: Implicit string-to-number conversion producing unexpected results
  • Null handling: NULL != NULL in SQL — joins on nullable columns lose records
# Quick debug pattern
# 1. Count at each stage
print(f"Input rows: {df_input.count()}")
print(f"After filter: {df_filtered.count()}")  
print(f"After join: {df_joined.count()}")  # Watch for fan-out or drop
print(f"After agg: {df_final.count()}")

# 2. Check for nulls in join keys
df_input.filter(col("join_key").isNull()).count()

# 3. Validate aggregation
df_final.select(sum("amount")).show()  # Compare with source total
